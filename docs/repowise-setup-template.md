# Repowise Setup Template

Step-by-step guide for setting up Repowise on any new project. Tested on Windows 11.

## 1. Prerequisites (one-time)

### Python 3.11+ via conda

Repowise requires Python >= 3.11. Use an isolated conda environment (do NOT upgrade base Python).

```batch
conda create -n repowise python=3.11 -y
conda activate repowise
pip install repowise==0.2.1
```

Verify:
```batch
C:\Users\anujd\miniconda3\envs\repowise\Scripts\repowise.exe --version
```

### Ollama + nomic-embed-text

```batch
:: Install Ollama from https://ollama.com if not already installed
ollama pull nomic-embed-text
ollama pull qwen3:8b
```

Verify Ollama is running:
```batch
curl http://localhost:11434/api/tags
```

## 2. Per-Project Setup

Replace `<PROJECT_PATH>` with the absolute path to your project (e.g., `C:\Users\anujd\Documents\01_AI\214_Chat_overlay_widget`).

### 2.1 Add .repowise/ to .gitignore

```batch
cd <PROJECT_PATH>
echo .repowise/ >> .gitignore
echo .mcp.json >> .gitignore
```

Verify both entries are in `.gitignore` (avoid duplicates if already present).

### 2.2 Initialize Repowise

```batch
set PYTHONUTF8=1
C:\Users\anujd\miniconda3\envs\repowise\Scripts\repowise.exe init --index-only
```

`--index-only` skips LLM calls during indexing, avoiding the 5-15% desync risk in alpha software.

Expected output: file count, symbol count, language stats, graph nodes/edges, elapsed time.

### 2.3 Write config.yaml

Create `<PROJECT_PATH>/.repowise/config.yaml`:

```yaml
llm:
  provider: ollama
  model: qwen3:8b
  embedding_model: nomic-embed-text

git:
  co_change_commit_limit: 500
  blame_enabled: true

dead_code:
  enabled: true
  threshold: 0.7
```

### 2.4 Configure MCP Server (project-scoped)

Create `<PROJECT_PATH>/.mcp.json`:

```json
{
  "mcpServers": {
    "repowise": {
      "command": "C:\\Users\\anujd\\miniconda3\\envs\\repowise\\Scripts\\repowise.exe",
      "args": ["mcp", "<PROJECT_PATH>"],
      "env": {
        "PYTHONUTF8": "1"
      }
    }
  }
}
```

Note: The path argument is positional (not `--path`). Use the absolute project path with forward or backslashes.

### 2.5 Generate CLAUDE.md

If the project already has a CLAUDE.md:

```batch
:: Step 1: Back up existing file
copy CLAUDE.md CLAUDE.md.backup

:: Step 2: Preview output (zero risk)
set PYTHONUTF8=1
C:\Users\anujd\miniconda3\envs\repowise\Scripts\repowise.exe generate-claude-md --stdout

:: Step 3: Add markers to bottom of existing CLAUDE.md (preserves all existing content)
:: Append these two lines:
:: <!-- REPOWISE:START -- Do not edit below this line. Auto-generated by Repowise. -->
:: <!-- REPOWISE:END -->

:: Step 4: Generate (fills between markers only)
C:\Users\anujd\miniconda3\envs\repowise\Scripts\repowise.exe generate-claude-md

:: Step 5: Verify original sections are intact
```

If no CLAUDE.md exists, just run `generate-claude-md` directly.

### 2.6 Verify with Doctor

```batch
set PYTHONUTF8=1
C:\Users\anujd\miniconda3\envs\repowise\Scripts\repowise.exe doctor
```

Expected: all checks pass. If issues found, try `doctor --repair`.

## 3. Daily Maintenance

Run the maintenance script (or these commands manually):

```batch
set PYTHONUTF8=1
set REPOWISE=C:\Users\anujd\miniconda3\envs\repowise\Scripts\repowise.exe

%REPOWISE% update
%REPOWISE% doctor
%REPOWISE% dead-code --safe-only
%REPOWISE% generate-claude-md
```

Or use the provided `scripts/repowise-maintenance.bat` as a template.

## 4. Available MCP Tools

Once configured, Claude Code exposes these tools:

| Tool | Status | Notes |
|------|--------|-------|
| get_overview | Working | Project summary, tech stack, entry points |
| get_context | Working | Code locations for specific targets |
| get_risk | Working | Change risk analysis |
| get_dependency_path | Working | Dependency chains between files |
| get_dead_code | Working | Unused code candidates |
| get_architecture_diagram | Working | Mermaid architecture diagrams |
| update_decision_records | Working | Add architectural decisions |
| repowise (meta) | Working | Tool usage instructions |
| get_why | BROKEN | Hangs 50% of calls. Use get_context + CLAUDE.md decisions instead |
| search_codebase | Partial | confidence_score always 1.0 (bug). Use relevance_score for ranking |

## 5. Known Issues and Workarounds

### PYTHONUTF8=1 Required

Windows uses cp1252 encoding by default. Repowise's Rich console output fails with `UnicodeEncodeError` without this env var. Always set `PYTHONUTF8=1` before running any repowise command.

### get_why() Hangs

Root cause: LLM synthesis call to Ollama can hang indefinitely. Workaround: use `get_context()` for code locations and read decisions from CLAUDE.md preserved sections. Optional mitigation: set `MCP_TOOL_TIMEOUT=15000` in Claude Code settings to convert hangs to recoverable timeouts.

### search_codebase() Confidence Always 1.0

Root cause: normalization formula `raw/max_score` guarantees top result = 1.0. Workaround: use `relevance_score` for ranking instead of `confidence_score`. For known targets, prefer `get_context()` over `search_codebase()`.

### repowise decision add is Interactive-Only

No `--non-interactive` flag exists. Document decisions in CLAUDE.md or docs/ instead, so they survive `.repowise/` rebuilds.

## 6. Troubleshooting

### UnicodeEncodeError on any command
```
set PYTHONUTF8=1
```

### Doctor reports corruption
```batch
C:\Users\anujd\miniconda3\envs\repowise\Scripts\repowise.exe doctor --repair
```

### Full reset (nuclear option)
```batch
:: .repowise/ is ephemeral -- safe to delete and rebuild
rmdir /s /q .repowise
set PYTHONUTF8=1
C:\Users\anujd\miniconda3\envs\repowise\Scripts\repowise.exe init --index-only
:: Re-apply config.yaml from step 2.3
:: Re-run generate-claude-md from step 2.5
```

Rebuild time: ~2-5 minutes for a 500-file project.

### MCP server won't connect
- Verify absolute path to repowise.exe is correct
- Verify PYTHONUTF8=1 is in the MCP env config
- Verify the project path argument is positional (not --path)
- Restart Claude Code after changing .mcp.json

### Ollama model not responding
```batch
ollama list          :: verify models are pulled
ollama ps            :: verify a model is loaded
curl localhost:11434 :: verify Ollama is running
```

### Dashboard chat/search with Ollama (instead of OpenAI API key)

Repowise dashboard uses a separate embedder from the MCP server. By default it prompts
for OpenAI or Gemini. To use Ollama's OpenAI-compatible API instead:

```batch
:: Set these env vars BEFORE running repowise serve
set REPOWISE_EMBEDDER=openai
set REPOWISE_EMBEDDING_MODEL=nomic-embed-text
set OPENAI_API_KEY=ollama
set OPENAI_BASE_URL=http://localhost:11434/v1
```

How it works:
- `REPOWISE_EMBEDDER=openai` tells the dashboard to use the OpenAI embedder class
- `REPOWISE_EMBEDDING_MODEL=nomic-embed-text` overrides the default `text-embedding-3-small`
  (which Ollama doesn't have) with `nomic-embed-text` (already pulled locally)
- `OPENAI_BASE_URL=http://localhost:11434/v1` redirects the OpenAI SDK to Ollama's
  OpenAI-compatible endpoint
- `OPENAI_API_KEY=ollama` satisfies the API key check (Ollama ignores the value)

After first launch with embedder enabled, trigger a full resync via the dashboard UI
or API (`POST /api/repos/<id>/full-resync`) to build the vector search index.

Source: discovered in `repowise/server/app.py` lines 55-75 (`_build_embedder` function)
and `repowise/core/providers/embedding/openai.py` line 72 (`REPOWISE_EMBEDDING_MODEL` env var).
